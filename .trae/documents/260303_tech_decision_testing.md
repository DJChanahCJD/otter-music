# 技术决策：自动化测试方案选型

**日期**: 2026-03-03
**状态**: 提案中
**作者**: Trae AI

## 1. 背景与约束
*   **现状**: 项目目前无任何自动化测试，依靠人工测试。
*   **目标**: 引入自动化测试，要求“简单高效”，避免过度设计。
*   **技术栈**: React, Vite, Capacitor, Shadcn-ui。
*   **约束**:
    *   **低维护成本**: 测试代码不应成为开发的负担。
    *   **高执行速度**: 测试应能快速反馈。
    *   **易于集成**: 需与现有 Vite 构建工具无缝协作。

## 2. 方案对比

### 方案 A: Vitest + React Testing Library (推荐)
*   **类型**: 单元测试 & 组件集成测试。
*   **优点**:
    *   **极速**: 基于 Vite，热更新和运行速度极快。
    *   **零配置**: 可复用 `vite.config.ts`，开箱即用。
    *   **兼容性**: 兼容 Jest API，学习成本低。
    *   **适用性**: 非常适合测试复杂的 Store 逻辑（如播放器状态）和工具函数。
*   **缺点**: 无法测试真实浏览器环境和原生设备特有功能。

### 方案 B: Playwright / Cypress
*   **类型**: 端到端 (E2E) 测试。
*   **优点**: 模拟真实用户操作，覆盖完整业务流程。
*   **缺点**:
    *   运行速度慢。
    *   环境依赖重（需要浏览器环境）。
    *   测试编写和维护成本较高（UI 变化容易导致测试失败）。

### 方案 C: Appium / Maestro
*   **类型**: 原生移动端测试。
*   **优点**: 能测试 Capacitor 原生插件交互。
*   **缺点**: 配置极复杂，运行极慢，严重违背“简单高效”原则。

## 3. 决策建议
**采用方案 A (Vitest)** 作为第一阶段的测试方案。

**理由**:
1.  **性价比最高**: 能够以最小的成本覆盖最核心的业务逻辑（数据处理、状态管理）。
2.  **符合现状**: OtterMusic 的核心复杂度在于播放状态管理和数据流，这些最适合用单元测试覆盖。
3.  **开发体验**: Vitest 的即时反馈能提升开发效率，而不是拖慢节奏。

## 4. 实施策略 (Roadmap)
1.  **基础建设**: 安装 Vitest 及相关库，配置运行环境。
2.  **核心覆盖**:
    *   优先测试 `src/lib/utils` 中的工具函数（如时间格式化、歌词解析）。
    *   测试 `src/store` 中的关键状态逻辑（如播放列表管理）。
3.  **组件测试**: 针对高复用的 UI 组件（如播放控制条）编写简单的渲染测试。
4.  **持续集成**: (后续) 将 `npm run test` 加入 CI 流程。

## 5. 强制规则 (Mandatory Rules)
1.  **工具函数必测**: `src/lib/utils` 下的新增纯函数必须包含单元测试。
2.  **不测 UI 细节**: 除非是核心交互组件，否则不为 UI 样式编写测试，避免频繁修改。
3.  **测试即文档**: 测试用例命名必须清晰表达业务意图（如 `it('should parse lrc lyrics correctly')`）。
