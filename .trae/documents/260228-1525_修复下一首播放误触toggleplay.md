## 目标
- 修复移动端曲目菜单（MusicTrackMobileMenu）点击“下一首播放”时，误触发所在歌曲行的点击播放逻辑，导致当前歌曲被 togglePlay（播放中变暂停/暂停变播放）的问题。

## 现象与复现路径（基于现有代码事实）
- 列表行点击播放入口在 [MusicTrackItem](file:///c:/Users/DJCHAN/SE/2_GithubProject/otter-music/src/components/MusicTrackItem.tsx#L75-L85)：行容器 `onClick={showCheckbox ? onSelect : onPlay}`。
- 移动端菜单在 [MusicTrackItem](file:///c:/Users/DJCHAN/SE/2_GithubProject/otter-music/src/components/MusicTrackItem.tsx#L153-L189) 内渲染，`onAddToNextPlay` 只是调用 store 的 `addToNextPlay(track)`，本身不应该影响播放状态。
- 菜单使用的 Drawer 组件 [drawer.tsx](file:///c:/Users/DJCHAN/SE/2_GithubProject/otter-music/src/components/ui/drawer.tsx#L48-L73) 通过 Portal 渲染 `DrawerContent`（`DrawerPrimitive.Portal`）。
- React 中 Portal 内部触发的事件会沿 React 组件树冒泡到父组件（即使 DOM 不在同一层级）。因此点击 `DrawerContent` 内按钮时，如果不在事件源头 stopPropagation，事件会冒泡到 `MusicTrackItem` 行容器，进而触发 `onPlay`。
- 当触发 `onPlay` 且点击的是当前正在播放的同一首歌时，业务逻辑会走到 `togglePlay()`（从而由 [GlobalMusicPlayer](file:///c:/Users/DJCHAN/SE/2_GithubProject/otter-music/src/components/GlobalMusicPlayer.tsx) 的播放控制 hooks 驱动 audio 暂停/播放切换）。

## 根因结论
- `MusicTrackMobileMenu` 仅在 Trigger 和外层 div 做了 `stopPropagation`（见 [MusicTrackMobileMenu](file:///c:/Users/DJCHAN/SE/2_GithubProject/otter-music/src/components/MusicTrackMobileMenu.tsx#L57-L71)），但 `DrawerContent` 是 Portal 渲染，内部按钮点击事件仍会沿 React 树冒泡到 `MusicTrackItem`，导致误触发行 `onClick`。

## 修改方案（最小改动）
- 在 `MusicTrackMobileMenu.tsx` 的 `DrawerContent`（或其内部最外层内容容器）增加事件拦截：
  - 使用 `onPointerDownCapture` + `onClickCapture` 统一 `stopPropagation()`，覆盖移动端 tap（pointer）与 click 两类触发。
  - 这样不需要逐个按钮补 stopPropagation，也能覆盖 `customActions` 插槽里的任意按钮，避免未来再复发。

## 验证方式
- 手动验证（移动端/小屏模拟优先）：
  1. 播放任意歌曲 A（确保处于播放中）。
  2. 打开歌曲 A 的“更多操作”抽屉，点击“下一首播放”。
  3. 期望：播放状态不改变（仍播放），仅队列顺序发生变化/提示 toast。
  4. 对非当前歌曲 B 重复步骤，期望：不触发播放切换/不开始播放 B，仅插入到下一首位置。
- 工程校验：

  - `npm run lint`
  - `npm run build`（可选但推荐，确保无编译问题）

## 风险与回滚
- 风险：对 DrawerContent 做 capture 阶段 stopPropagation 可能会影响“点击抽屉内部导致外部某些点击处理”的场景；但本项目期望抽屉内部操作不影响列表行点击，属于符合预期的隔离。
- 回滚：撤销 `DrawerContent` 上新增的捕获事件处理即可恢复原行为。
